# AGENT.md

このリポジトリでの実装ルールを定義する。

## 適用範囲

- このプロジェクトは Astmend の実装に関するものです。
- web フレームワーク、API サーバー、DB、フロントエンド UI は対象外です。
- そのため、`api/modules/`、`src/modules/`、`shared/schemas/`、TanStack Router、TanStack Query、Drizzle、Storybook、Pencil に関する規則はこのリポジトリには適用しません。
- 不要な抽象化や将来機能の先回り実装は行わない。

## 1. モジュラー・モノリス構成

- このプロジェクトでは web アプリの層構造は採用しない。
- 必要な場合のみ、責務単位で `src/` 配下を分割する。
- 分割は KISS と SRP を優先し、過剰な階層化は避ける。

## 2. API 通信

- web API 通信は実装しない。
- `fetch` や RPC クライアントはこのプロジェクトでは不要。

## 3. 基本技術ルール

- 実装対象が TypeScript ベースの AST パッチエンジンであることを前提にする。
- 必要な依存関係だけを追加し、過剰な汎用化をしない。
- 変更は最小差分で行う。

## 4. ソフトウェア設計の基本原則

- KISS: 極力シンプルにする。
- YAGNI: 推測に基づく抽象化を避ける。
- DRY: 重複は避けるが、無理な共通化はしない。
- SRP: モジュールと関数は単一責任に保つ。

## 5. ベストプラクティス・心得

- 可読性を最優先する。
- 早期リターンを徹底する。
- エラーを握り潰さず、適切に伝播させる。
- 触った周辺コードも必要に応じて小さく改善する。
- オーバーエンジニアリングを避ける。

## 6. テストのルール

- Flaky Tests を作らない。
- UI スナップショットテストは行わない。
- 実装詳細ではなく、入出力の振る舞いをテストする。
- カバレッジを目的化しない。

## 7. デザインシステム / CSS 変更のルール

- このプロジェクトでは CSS 変更やデザインシステム変更は原則扱わない。
- したがって Tailwind、CSS 変数、Storybook に関するルールは適用外。

## 8. 個人情報保護とログ

- 個人情報・認証情報をログに残さない。
- URL は原則 path のみを扱い、query は記録しない。
- デバッグでも機微情報の生値は出力しない。

## 9. 実装時の優先事項

- まず `plan.md` と `implementation-plan.md` を参照する。
- 実装時はこの `AGENT.md` を優先し、不要な web 系の前提を持ち込まない。
- 既存の計画に反する変更が必要な場合は、その理由を明示してから進める。
